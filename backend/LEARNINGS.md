
## L26 — Scoring must run at temperature=0 (grading task; default 1.0 added tier-flipping noise)
**Gap:** `score_role`'s Claude call set no `temperature`, so it ran at the default 1.0. Scoring is a grading task — the same JD should score the same number — but at temp 1.0 it carried ±several points of run-to-run noise, enough to flip a role across the Good/Strong boundary (79/80). Surfaced when Sam compared two near-identical Google "Strategy and Operations Manager, GTM/Go-to-Market" roles; the noise muddied a genuine fit difference (the two JDs actually differ in quals — one is GTM-systems/data-strategy, the other finance/IB/sales-ops flavored — but variance made the comparison unreadable).
**Guard:** scoring call now passes `temperature=0`. Verified deterministic (same role scored 83 twice). selfcheck **L26** asserts `temperature=0` is present in scoring.py's messages.create. Net: a role's tier no longer depends on which run scored it, so real fit differences (and calibration changes) are the only thing that moves a score.

## L25 — Amazon fetch silently collapsed to ~0 (httpx gzip decompressor reuse)
**Gap:** `fetch_amazon_jobs` reuses one `httpx.AsyncClient` across its per-query loop. amazon.jobs returns gzip-encoded responses, and reusing the client triggered httpx's "cannot use a decompressobj multiple times" — EVERY query raised, so the function returned ~2 jobs instead of ~74. Amazon discovery (Sam's current employer, internal-transfer pipeline) was effectively dead, and it failed silently (caught per-query, logged, empty result) — only surfaced by a coverage audit. Throughput-critical because Amazon roles score on a lower bar (70) and lifted internal-transfer context.
**Guard:** Amazon request headers now send `Accept-Encoding: identity` (request uncompressed → no decompressor → no reuse bug). Verified: 2 → 74 jobs. selfcheck **L25** asserts the header is present in fetch_amazon_jobs. If a future ATS client loops requests on a shared client and hits the same error, apply the same fix (or use a fresh client per request).

## L24 — Discovery role-keyword filter was too narrow (dropped genuine targets pre-scoring)
**Gap:** `filter_jobs_for_profile` gates every board role through `ROLE_KEYWORDS` before scoring — comprehensiveness depends entirely on that list. A coverage audit (board-matched vs tracked, then false-negative analysis) found genuine target roles being dropped because their title family wasn't a keyword: "Business Strategy ..." (Google "Business Strategy and GTM Partner Operations Lead"), "Business Operations & Strategy" (OpenAI "GTM Business Operations & Strategy Lead" — also failed on "&" vs "and"), and partnerships roles ("Amazon GTM Partnership, Startups" — which Sam applied to — "Startup Partnerships Lead", "Strategic Partnerships Lead") even though **AI Partnerships is an explicit target_role_type**. Every dropped role is an application Sam never sees — directly caps throughput.
**Guard:** ROLE_KEYWORDS now includes the business-strategy / business-operations / gtm-operations families and SPECIFIC partnership families (ai/gtm/strategic/startup/frontier/platform partnership — kept narrow to exclude channel/SI/event/federal sales-partnerships). `_matches_role_keywords` normalizes " & " → " and " so "Ops & Strategy" matches the "...and strategy" keywords. The EXCLUDE list still removes eng/design/CS/sales/finance/sourcing noise, so precision holds (verified: real targets match, "Strategic Finance/Sourcing/Pricing", "Federal Partnerships", BDR/SDR stay excluded). selfcheck **L24** verifies the keyword families exist + the & normalization + behavioral match/no-match cases. Principle: be inclusive at the discovery FILTER (coarse net, don't miss), precise at SCORING (the calibrated ranker decides tier) — a dropped role can't be recovered downstream, a low-fit role just scores low.

## L23 — Role dedup must key on (title, location), not title alone (generic-title collapse)
**Gap:** Discovery deduped new roles by (company, title) — added to collapse the SAME role posted under multiple ATS requisition IDs. But generic titles are reused for genuinely DIFFERENT roles: Google posts several distinct "Strategy and Operations Lead" roles across orgs/levels/locations (e.g. Core/Advanced/Sunnyvale vs GCS-SMB/Mid/Mountain View). Title-only dedup silently dropped all but the first, so distinct target roles never got tracked or scored — found only because Sam pasted them from LinkedIn and asked us to validate. A Strong (81) "S&O Lead, Core" role was being missed entirely.
**Guard:** `_dedup_key(title, location, raw_jd)` in discovery.py collapses a link only when title + normalized location (multi-city sorted) + JD fingerprint ALL match — i.e. the same posting under multiple requisition IDs. Used by BOTH `discover_via_ats` and `discover_via_linkedin` (DB step + within-batch step). Kept-as-separate (so Sam can apply to each): same title + different location (Seattle vs SFO); same title + same location + different JD (two different MountainView PMs). Collapsed: same title + same location + same JD (true repost). URL dedup (step 1) still prevents exact re-insertion on re-runs. **Why JD too (added after Sam pushed on it):** title+location alone still wrongly collapses two genuinely-different roles that share a common title+location (e.g. multiple "Product Manager" in Mountain View) — the JD fingerprint is what makes dedup collapse only provably-identical postings, never a distinct applyable link. selfcheck **L23** verifies both paths use `_dedup_key` (no title-only sets) and the behavior (diff location → keep; same title+location+diff JD → keep; identical repost w/ reordered cities → collapse).

## L22 — Greenhouse own-domain boards carry the job id in the query string (URL collapse)
**Gap:** Databricks embeds Greenhouse on its own domain with the job id in the QUERY string (`databricks.com/company/careers/open-positions/job?gh_jid=123`). The discovery URL-normalizer (discovery.py) strips query strings for dedup hygiene, collapsing **all** Databricks postings to one identical path — so only 1 of 27 ever inserted (the rest looked like URL-duplicates) and the stored "View Original" link was a dead generic careers page. Boards where the id is already in the path (Anthropic, DeepMind) were unaffected.
**Guard:** `fetch_greenhouse_jobs()` now falls back to the canonical Greenhouse-hosted URL `https://job-boards.greenhouse.io/{slug}/jobs/{id}` whenever `absolute_url` doesn't already contain `/jobs/{id}` — id stays in the path, so normalization keeps each posting unique and the link resolves to the specific job. selfcheck **L22** verifies the fallback exists. If a new ATS client emits ids in the query string, apply the same id-in-path treatment.
