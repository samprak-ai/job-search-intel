
## L23 — Role dedup must key on (title, location), not title alone (generic-title collapse)
**Gap:** Discovery deduped new roles by (company, title) — added to collapse the SAME role posted under multiple ATS requisition IDs. But generic titles are reused for genuinely DIFFERENT roles: Google posts several distinct "Strategy and Operations Lead" roles across orgs/levels/locations (e.g. Core/Advanced/Sunnyvale vs GCS-SMB/Mid/Mountain View). Title-only dedup silently dropped all but the first, so distinct target roles never got tracked or scored — found only because Sam pasted them from LinkedIn and asked us to validate. A Strong (81) "S&O Lead, Core" role was being missed entirely.
**Guard:** `_dedup_key(title, location)` in discovery.py keys dedup on title + normalized location (multi-city sorted), used by BOTH `discover_via_ats` and `discover_via_linkedin` (DB step + within-batch step). Same title + same location → collapse (same role, multiple req IDs); same title + different location → keep both (different roles). URL dedup (step 1) still prevents exact re-insertion. selfcheck **L23** verifies both paths use `_dedup_key` (not the old title-only set) and checks the behavior (same-title/diff-location → distinct keys; same-title/same-location incl. reordered cities → same key). Trade-off accepted: the same role posted in two genuinely different locations as separate req IDs may now appear twice — a mild, skippable dup, strictly better than dropping a distinct role.

## L22 — Greenhouse own-domain boards carry the job id in the query string (URL collapse)
**Gap:** Databricks embeds Greenhouse on its own domain with the job id in the QUERY string (`databricks.com/company/careers/open-positions/job?gh_jid=123`). The discovery URL-normalizer (discovery.py) strips query strings for dedup hygiene, collapsing **all** Databricks postings to one identical path — so only 1 of 27 ever inserted (the rest looked like URL-duplicates) and the stored "View Original" link was a dead generic careers page. Boards where the id is already in the path (Anthropic, DeepMind) were unaffected.
**Guard:** `fetch_greenhouse_jobs()` now falls back to the canonical Greenhouse-hosted URL `https://job-boards.greenhouse.io/{slug}/jobs/{id}` whenever `absolute_url` doesn't already contain `/jobs/{id}` — id stays in the path, so normalization keeps each posting unique and the link resolves to the specific job. selfcheck **L22** verifies the fallback exists. If a new ATS client emits ids in the query string, apply the same id-in-path treatment.

## L20 — Amazon Principal+ roles are a level-up, not a realistic internal transfer
**Gap:** The internal-transfer scoring lift (lifts the big-co 84 cap, h1b=100, grade-on-fit) was inflating Amazon **Principal** roles to Perfect/Strong (95s) — but Principal (L7) is a level up from Sam's band, and internal transfers rarely clear a level jump, so those aren't realistic targets. They were crowding the match list.
**Guard:** `AMAZON_LEVEL_UP_RE` in ats_clients.py (`\b(principal|director|vice president|vp)\b`) makes `fetch_amazon_jobs()` skip level-up titles at the source, before scoring. selfcheck **L20** (static) verifies the filter exists + the regex blocks Principal/Director and passes Senior/PMT/Worldwide-Specialist; **L20 db** check asserts no Amazon Principal+ role is still tracked. Existing offenders were purged when the rule landed. If Sam's band changes (e.g. he becomes Principal), relax the regex accordingly.

## L17 — Amazon-internal performance language must stay off external artifacts

It passed the audit because it is *true* (groundedness ok) and well-formatted — but it's Amazon-only insider detail that means nothing to an external reviewer and signals internal-review copy-paste. **Guard:** reviewer.py block (e) flags Amazon-internal performance terms ("exceeds high bar", "annual review", "annual performance rating", "forte goals", "promo doc") as **must_fix** when company != Amazon (advisory when company unknown); silent for Amazon. selfcheck L17 verifies the rule text + behavior (must_fix on external, silent on Amazon). Keep the fact in profile.json/locked_facts/sam-profile.md (true; needed for Amazon packets) — just keep it out of external resumes/letters.
